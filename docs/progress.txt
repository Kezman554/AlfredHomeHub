2026-07-15
- Added GET /daily-schedule to the Vault API (src/vault_api/routers/schedule.py):
  today's planned items from the current ISO week's plan file at
  1-daily/reviews/YYYY-Wnn-plan.md. Returns [{task, period}] where period is
  "am"/"pm"/null.
- Parser + Vault.daily_schedule_items() live in vault.py: derives the plan file
  from today via isocalendar(), finds today's day heading ("### Tue 14 Jul"),
  and reads its bullets. Period comes from an inline "AM:"/"PM:"/"Evening:"/
  "Night:"/"Pre-breakfast:" prefix, else an AM/PM subheading grouping, else null
  (morning-ish -> am; afternoon/evening/night -> pm). AM/PM split is optional:
  plain untagged bullets return period null, not an error.
- Robust by design: no plan file this week, no section for today, no bullets, or
  an unreadable vault all return [] with a 200, never a 500.
- Tests in tests/test_schedule.py (11, all passing): inline prefixes, AM/PM
  subheadings, untagged bullets, today absent -> [], missing file -> [],
  unreadable root -> [], trailing "## Notes" section doesn't leak, shape is
  {task, period} with period only am/pm/null, ISO week path formatting. Also
  smoke-tested the live endpoint via TestClient against a fixture plan.
- NOT yet run on the Pi — needs a pull + `docker compose up -d --build`, then
  curl 192.168.1.100:8200/daily-schedule from the laptop.

  path (locked git transaction — shared flock, pull --ff-only, surgical edit,

  a discovered family of lists. No second write path built.
- Discovery is by content, not location: GET /shopping returns every vault
  .md tagged both life/shopping and status: active, wherever it lives (
  6-life/shopping/ is just where CREATE LIST puts new ones). Per list:
  vault-relative path as id, H1 (or filename) as title, total/unticked
  counts. Item lines have no dates (unlike rolling-todo) and, unlike the
  chalkboard, reads include ticked items too — they stay visible until swept.
- New Vault methods: shopping_lists/_scan_shopping_lists (discovery),
  shopping_list_items (one list's summary + items), add/tick/drop_shopping_
  item, create_shopping_list, and a combined sweep() replacing sweep_ticked.
  _resolve_shopping_list guards every list_id against path traversal and
  against targeting a list that isn't currently tagged+active (404s with
  current discovery, same "stale key" pattern as chalkboard's item 404s).
- Router src/vault_api/routers/shopping.py: GET/POST /shopping (discovery,
  create), GET/POST /shopping/{list_id} (read, add), .../tick, .../drop.
  {list_id:path} is a greedy match, so /tick and /drop are registered before
  the plain add route, or a POST to ".../fitness.md/tick" would be swallowed
  by add with list_id="...fitness.md/tick" instead of reaching tick.
- Add inserts the new checkbox line right after the list's last existing
  item (never at the very end of a file with trailing prose/table sections
  after the items — that fell out of "after the last item" for free); an
  item-less freshly-scaffolded list falls back to just after its
  "**Format:**" line, before whatever section follows.
- CREATE LIST kebab-cases the display name into the filename under
  6-life/shopping/ (422 on a name with nothing kebab-able, 409 if the file
  exists already) and scaffolds vault house-style frontmatter + H1 + Format

- Sweep extension: POST /chalkboard/sweep (same endpoint, same script,
  scripts/sweep-todo.sh untouched) now sweeps every discovered shopping list
  in the same commit as the rolling to-do. Response keeps swept/count/items
  as rolling-todo-only for existing callers; adds shopping_swept/
  shopping_count. Only when at least one shopping item was swept, one

2026-07-20
- Inbox capture endpoint pair (src/vault_api/routers/inbox.py): POST /inbox
  writes 0-inbox/YYYY-MM-DD-HHMM-slug.md holding the posted text verbatim —
  no frontmatter, no headers — matching the vault's inbox convention (a few
  plain bullets or prose, ephemeral). GET /inbox lists every 0-inbox/ .md as
  [{filename, content}], newest first.
- POST body is text/plain, not JSON: captures come from voice and shortcuts,
  where wrapping a sentence in JSON is friction. Empty or >10k bodies 422.
- Slug is the first 6 words kebab-cased, capped at 48 chars, falling back to
  "note" when nothing kebab-able survives (the timestamp still makes the name
  unique). Commit message "alfred api: inbox capture '<slug>'", authored as
  Alfred <alfred@alfred.local>, same as T3/T4.
- Reuses the existing write machinery unchanged — shared flock, pull
  --ff-only, write, commit, push, hard-reset on failure. The only change to
  existing code is _write_inbox_capture (previously the shopping sweep's
  private helper) taking a slug and, on a name collision, suffixing -2, -3
  rather than overwriting: no dedupe or merge, every POST is a new file, but
  two captures in the same minute never clobber each other. Sweep behaviour
  is unchanged.
- Listing sorts by filename descending, not mtime: names lead with the
  timestamp, and a fresh clone on the Pi gives every file the same checkout
  mtime, so the name is the only capture time that survives git.
- Deliberately not built: no tag parsing/routing (#todo, #idea), no
  delete/edit/triage endpoint (inbox notes are triaged by hand in a vault
  session), no dedupe or merge.
- Tests: tests/test_inbox.py (25, all passing; 83 total) — slug cases
  (word cap, length cap, punctuation, fallback), the write transaction
  against a real local git origin+clone (raw content with no frontmatter,
  filename shape, commit message + Alfred authorship, pushed to origin,
  clean tree after, identical captures produce two files, CRLF normalised,
  empty rejected), listing (newest first, non-.md ignored, missing dir ->
  [], round trip), and four over HTTP via TestClient proving the text/plain
  body binds (a wrong media_type would silently 422 every capture).
  Write tests bypass the POSIX-only flock via monkeypatch, as in T5.
